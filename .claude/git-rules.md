# Git 관리 규칙

## 🚫 Git Push 정책 (절대 규칙)

**사용자 승인 없이 Git Push 절대 금지**

---

## 📋 작업 프로세스

```
1. 코드 수정 완료
   ↓
2. 로컬 테스트 (npm run dev)
   ↓
3. 사용자에게 확인 요청
   ↓
4. 사용자 승인 후에만 Git Push
```

---

## ✅ 승인 대기 메시지

코드 수정 완료 후 **반드시** 다음과 같이 안내:

```
"수정 완료했습니다!

로컬 서버(localhost:3000)에서 확인해주세요.
문제없으면 '푸쉬해' 또는 '배포해' 라고 말씀해주세요!"
```

---

## 🎯 승인 키워드

사용자가 다음과 같이 말하면 **Git Push 진행**:

- ✅ "푸쉬해"
- ✅ "커밋해"
- ✅ "배포해"
- ✅ "Git에 올려"
- ✅ "좋아"
- ✅ "문제없어"
- ✅ "오케이" / "OK"
- ✅ "괜찮아"

---

## ❌ 거부 키워드

사용자가 다음과 같이 말하면 **수정 진행**:

- ❌ "수정해"
- ❌ "다시"
- ❌ "안돼"
- ❌ "문제있어"
- ❌ "바꿔"
- ❌ "고쳐"

---

## 💻 Git 명령어 실행

승인 후에만 다음 순서로 실행:

```bash
git add .
git commit -m "feat: [작업 내용]

🤖 Generated with [Claude Code](https://claude.com/claude-code)

Co-Authored-By: Claude <noreply@anthropic.com>"
git push
```

---

## ⚠️ 예외 사항

다음 경우에만 자동 커밋 허용:

1. 사용자가 명시적으로 **"자동으로 푸쉬해도 돼"** 라고 허락한 경우
2. 긴급 버그 수정으로 즉시 배포가 필요한 경우 (단, 사용자에게 사전 확인 필요)

---

## 📝 실제 사용 예시

### 예시 1: 정상 프로세스

```
Claude: "다크모드 토글 버튼 수정 완료했습니다!
        로컬 서버에서 확인해주세요. 문제없으면 '푸쉬해' 라고 말씀해주세요!"

User: "잘 되네, 푸쉬해"

Claude: "Git에 푸쉬하고 Vercel 배포 진행하겠습니다!"
        [git add . && git commit && git push 실행]
```

### 예시 2: 수정 요청

```
Claude: "대시보드 카드 수정 완료했습니다!
        로컬 서버에서 확인해주세요."

User: "색상이 너무 진한데? 바꿔"

Claude: "알겠습니다. 색상을 더 밝게 수정하겠습니다."
        [수정 진행, Git Push 하지 않음]
```

---

## 🎯 핵심 원칙

1. **수정 완료 = 승인 대기**, Git Push 아님
2. **승인 키워드 확인 후에만** Git Push
3. **의심스러우면 물어보기** - "푸쉬해도 될까요?"
